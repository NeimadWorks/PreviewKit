# PreviewKit fixtures

A small, hand-curated set of files exercising every PreviewKit renderer.
Generated by `Fixtures/scripts/build-fixtures.sh` (idempotent — overwrites
in place, no external downloads, only system tools).

## Build

```sh
bash Fixtures/scripts/build-fixtures.sh
```

Outputs into `Fixtures/files/`. Re-run after pulling — the SQLite fixture
in particular is regenerated each run.

## Known runtime issues — DO NOT LOSE

These were observed when wiring the fixtures into Canopy via the
`PreviewKitBridgePlugin`. Selection in the file browser triggered
the inspector for each fixture. Most renderers were never confirmed
green in their host context.

### CRASH — MediaRenderer on audio fixture (P0)

Selecting `sample.aiff` in Canopy crashes the host with `SIGABRT` /
`abort()` from `getSuperclassMetadata` in `_AVKit_SwiftUI`.

**Stack (top frames)**

```
3   libsystem_c.dylib            abort + 148
4   libswiftCore.dylib           swift::fatalErrorv
5   libswiftCore.dylib           getSuperclassMetadata + 828
6   libswiftCore.dylib           _swift_initClassMetadataImpl + 100
7   _AVKit_SwiftUI               + 6516 (instantiateGenericMetadata)
…
27  SwiftUI                      ViewResponderFilter.init(inputs:view:)
```

Concurrent thread is in `MediaAnalyzer.waveformSamples` (line 194)
running `AVAssetReader.startReading()`, and Thread 3 is in
`tcc.preflight.kTCCServiceMicrophone` — the audio chain is asking
the host for **microphone** access, which an inspector should never do.

**Root cause** : `MediaRenderer` (originally Cairn-side) renders
`AVKit.VideoPlayer` for every media kind (audio + video). For audio
files this is wrong on two counts:

1. The AVKit_SwiftUI view tree is heavier and exposes a metadata path
   that fails on macOS 26.4.1 in the Canopy host.
2. AVAssetReader pulls in CoreAudio's mix server, which probes TCC
   for microphone before any audio is actually played back.

**Suggested fix** : split MediaRenderer into two paths.

- Audio (`.audio` kind) : draw the existing `WaveformView` only.
  Use a lightweight `AVAudioPlayer` for playback if needed; do NOT
  instantiate `VideoPlayer`. Avoid `AVAssetReader` outside an explicit
  user "build waveform" action — for the smoke pass, just show
  duration + format from `AVURLAsset.load(.duration, .tracks)`.
- Video (`.video` kind) : keep `VideoPlayer`.

Then file an upstream report for the AVKit_SwiftUI metadata crash on
macOS 26 — that's likely a SwiftUI bug, not ours, but the workaround
above sidesteps it.

### Pry spec can't reach the fixtures via Go-To-Path

`Tests/Pry/Flows/13-previewkit-inspectors.md` (in the Canopy repo)
opens `Cmd+Shift+G`, sends `Cmd+A` to select-all, types the path, and
sends Return. Observed behaviour : the field's existing value is
**not** replaced — typed text is dropped, navigation never happens,
the spec fails at step 8 (`wait_for activeTab.path matches …`).

The same pattern works in `11-sysadmin-jump.md` against `/Applications`,
so the bug is path- or focus-specific. Investigate before shipping the
inspector spec :

- Is the `Cmd+A` reaching the field editor when the GoToPath sheet is
  freshly opened? Compare with the working sysadmin-jump path.
- Try `Cmd+A` → `key: "delete"` (forward delete) → type, vs
  triple-click on the field, vs `key: "cmd+shift+left"` then delete.
- Verify Canopy's GoToPath text field actually selects-all on Cmd+A
  (the SwiftUI `TextField` field-editor behaviour can vary in sheets).

A fixture copy lives at `~/Documents/canopy-fixtures/` precisely so
the spec can reach the directory through the sidebar Documents entry,
bypassing GoToPath entirely. The 13-spec wasn't yet rewritten to use
that path.

### File rows expose `AXStaticText`, not `AXButton` (informational)

Canopy's file list rows expose `AXStaticText` with `label="File: <name>"`
or `label="Folder: <name>"`. Targeting `AXButton` with the filename does
not resolve. The 13-spec already uses the correct form
(`{ role: AXStaticText, label: "File: sample.md" }`), but a previous
version did not — easy regression to repeat.

Even with the correct target, a single click on the static text does
**not** update `AppState.activeTab.selection.count`. Likely a SwiftUI
hit-test issue : the static text is the visible label but the row's
gesture recognizer is on a parent. Worth investigating with `dump_focus`
+ point-based clicks before assuming it's broken.

### Untested renderers (every other one)

None of the per-fixture content assertions in `13-previewkit-inspectors.md`
have ever passed end-to-end. Every renderer below should be verified
on its fixture and have its expected-content assertion tightened or
relaxed based on what actually appears in the AX tree:

- `MarkdownRenderer` — assert "Markdown fixture" appears
- `SourceCodeRenderer` — assert "PreviewID" appears
- `DataRenderer` — JSON / YAML / CSV / XML / plist
- `PatchRenderer` — assert "sample.swift" appears
- `CalendarRenderer` — assert "fixture meeting" appears
- `ContactRenderer` — assert "Camille Dupont" appears
- `WebShortcutRenderer` — assert "neimad.fr" / "github.com"
- `SQLiteRenderer` — assert "Safari History" (fingerprint check)
- `ImageRenderer` (PNG/JPEG/HEIC) — currently only checks filename
- `FontRenderer` — currently only checks filename
- `ArchiveRenderer` — currently only checks filename
- `AppBundleRenderer` — assert "Calculator" appears
- `GPGRenderer` — assert "PGP" appears
- `MobileProvisionRenderer` — assert "Provisioning" appears

Many of these will fail because:
- The renderer hasn't loaded yet (timing — `sleep` is too short)
- The renderer rendered the data into a non-AX-readable form
  (e.g. `Canvas`, custom drawing, HTML preview, etc.)
- The label we're asserting on isn't actually rendered

This needs a session of "click each fixture, dump the inspector AX
tree, write the assertion against what actually appears." Plan
~10 minutes per fixture × 16 = 3-4 hours of focused work.
